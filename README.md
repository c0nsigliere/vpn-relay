# VPN Relay & WireGuard Cascade вҖ” Ansible

Production-ready Ansible repository for Ubuntu 22.04/24.04 supporting two VPN forwarding modes:

| Mode | Playbook | Purpose |
|------|----------|---------|
| **WireGuard cascade** | `playbooks/wg_cascade.yml` | Client вҶ’ A вҶ’ B вҶ’ Internet (AWG/plain WireGuard) |
| **XRay L4 relay** | `playbooks/relay.yml` | Transparent TCP forward A вҶ’ B (XRay/VLESS) |
| **XRay VLESS+Reality** | `playbooks/xray.yml` | Native XRay server on B (VLESS+Reality :8443) |
| **Full stack** | `playbooks/stack.yml` | Single entrypoint: maintenance вҶ’ swap вҶ’ cascade вҶ’ xray вҶ’ relay вҶ’ verify вҶ’ bot (optional) |

All can run simultaneously. The XRay relay on A forwards TCP to B where the native XRay server handles VLESS+Reality.

## Architecture

### WireGuard Cascade (primary вҖ” AWG/WireGuard)

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ    wg-clients      в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ  XRay VLESS+Reality  в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ

```

- Client has its **own WireGuard keypair** (generated by the Telegram bot)
- Server A terminates the client tunnel; **XRay TPROXY** intercepts all wg-clients traffic
- **Why XRay TPROXY:** Russian ISPs (TSPU) block outbound UDP вҖ” direct WireGuard to B is impossible.
  TPROXY forwards the decrypted client packets via VLESS+Reality, preserving original destination.
- Server A's **SSH and main default route are never affected** (TPROXY only applies to wg-clients)
- **Server B has no WireGuard** вҖ” XRay freedom outbound creates connections directly to the internet

### XRay L4 Relay + VLESS+Reality (parallel вҖ” TCP/VLESS)

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ         в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ         в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ  Client  в”Ӯ в”Җв”Җв”Җв”Җв”Җв”Җ> в”Ӯ   Server A       в”Ӯ в”Җв”Җв”Җв”Җв”Җв”Җ> в”Ӯ   Server B       в”Ӯ
в”Ӯ          в”Ӯ         в”Ӯ   (Relay)        в”Ӯ         в”Ӯ XRay native svc  в”Ӯ
в”Ӯ B's keys в”Ӯ         в”Ӯ DNAT+MASQ (TCP)  в”Ӯ         в”Ӯ VLESS+Reality    в”Ӯ
в”Ӯ A's IP   в”Ӯ         в”Ӯ No keys/decrypt  в”Ӯ         в”Ӯ   :8443          в”Ӯ
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ         в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ         в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
     :443                                               :8443
```

- Client uses **Server B's keys** (relay is transparent)
- Server A forwards raw TCP to Server B unchanged (`roles/relay/`)
- Server B runs native XRay VLESS+Reality (`roles/xray_server/`)
- Managed by `playbooks/relay.yml` (A) + `playbooks/xray.yml` (B)
- **From Russia**: clients connect to Server A (Russian IP) to bypass DPI вҖ” direct connections to Server B are filtered (see DPI notes below)

## Golden Path: Fresh Install (Cascade + Relay)

### 1. Install dependencies

```bash
# On Ubuntu/Debian (control node)
sudo apt install ansible sshpass wireguard-tools -y

# Install Ansible collections
ansible-galaxy collection install -r requirements.yml
```

### 2. Copy and edit inventory

```bash
cp inventory/inventory.ini.example inventory/inventory.ini
```

Fill in real IPs for `server-a` and `server-b`.

### 2a. Bootstrap SSH keys (first-time setup from root password)

If your servers currently use root password login, run this once to switch to key auth:

```bash
# Step 1: push your SSH public key (uses ansible_password from inventory)
ansible-playbook playbooks/bootstrap_ssh.yml --tags push_key \
  -e "ssh_public_key_file=~/.ssh/id_rsa.pub"

# Verify key login works manually:
ssh root@<server-a-ip>
ssh root@<server-b-ip>

# Step 2: disable password authentication on both servers
ansible-playbook playbooks/bootstrap_ssh.yml --tags harden
```

After step 2 succeeds, **remove `ansible_password` lines from `inventory/inventory.ini`**.
The servers will reject password logins going forward.

### 3. Copy and edit group_vars

```bash
# Shared vars (REQUIRED вҖ” set server_b_public_ip here)
cp inventory/group_vars/all.yml.example inventory/group_vars/all.yml

# Cascade vars (review defaults вҖ” usually fine as-is)
cp inventory/group_vars/wg_cascade.yml.example inventory/group_vars/wg_cascade.yml

# Relay vars (review ports вҖ” usually fine as-is)
cp inventory/group_vars/relay_servers.yml.example inventory/group_vars/relay_servers.yml

# XRay server vars (review Reality settings вҖ” usually fine as-is)
cp inventory/group_vars/xray_servers.yml.example inventory/group_vars/xray_servers.yml
```

Edit `all.yml` вҖ” the **one required field**:
```yaml
server_b_public_ip: "YOUR_SERVER_B_PUBLIC_IP"
```

### 4. Optional: add swap on low-RAM servers

```bash
ansible-playbook playbooks/maintenance_add_swap.yml
```

### 5. Deploy

```bash
# Full stack (recommended вҖ” runs everything in correct order):
ansible-playbook playbooks/stack.yml

# Or deploy individual subsystems:
ansible-playbook playbooks/wg_cascade.yml    # WireGuard cascade (A + B)
ansible-playbook playbooks/xray.yml          # XRay VLESS+Reality (B)
ansible-playbook playbooks/relay.yml         # TCP relay (A)
```

### 6. Verify

```bash
ansible-playbook playbooks/verify_all.yml
```

Note: `stack.yml` already includes verification as the final step.

### 7. Add clients

Clients are managed exclusively through the **Telegram bot** (or TMA web app).
The bot writes to its SQLite DB and rebuilds XRay `config.json` automatically.
WireGuard peers are added via SSH to Server A.

> **Why no Ansible playbooks for client provisioning?**
> The bot's DB is the single source of truth. Clients created outside the bot
> would become "phantoms" вҖ” invisible to monitoring, quotas, and suspend/resume.

See step 8 (bot) or step 9 (TMA) below to set up the admin interface.

### 8. Deploy Telegram bot (recommended)

The bot runs on Server B and provides a Telegram-based admin panel for real-time client management.

```bash
# Deploy with bot credentials:
ansible-playbook playbooks/deploy_bot.yml \
  -e "bot_telegram_token=123456:ABC bot_admin_id=987654321"

# Or include in full stack (bot step only runs when bot_telegram_token is set):
ansible-playbook playbooks/stack.yml \
  -e "bot_telegram_token=123456:ABC bot_admin_id=987654321"
```

**Bot features:**
- Add/remove/suspend WireGuard and XRay clients from Telegram
- Send client configs (WG `.conf` file, VLESS URIs + QR codes) directly in chat
- Traffic graphs (chartjs PNG) per client
- Server status (CPU/RAM/uptime for A and B)
- Auto-suspend expired clients (TTL support)
- Health alerts if Server A goes unreachable
- Enriched update alerts вҖ” lists which packages need updating with changelogs and optional AI summaries (CVEs for security, brief descriptions for regular). Set `OPENAI_API_KEY` for AI summaries; degrades gracefully without it
- Client IP tracking (WG endpoint + XRay access log) with ISP lookup (ip-api.com)

**Required:** Get a bot token from [@BotFather](https://t.me/BotFather) and your numeric Telegram user ID.

**To remove the bot:**

```bash
# Dry-run first
ansible-playbook playbooks/remove_bot.yml --check

# Remove bot, keep SQLite database
ansible-playbook playbooks/remove_bot.yml -e "bot_remove=true"

# Remove bot and database
ansible-playbook playbooks/remove_bot.yml -e "bot_remove=true bot_remove_data=true"
```

The safety gate (`bot_remove=true`) is required to prevent accidental removal. The database at `/var/lib/vpn-bot/data.db` is preserved by default.

### 9. Deploy TMA Web App (optional)

The TMA is a React SPA inside Telegram вҖ” admin manages clients via a full web UI instead of inline keyboards.

**Prerequisites:**
- A domain pointing to Server B's IP (e.g. `vpn.example.com`)
- `tma_certbot_email` for Let's Encrypt registration

```bash
# Deploy nginx + certbot + rebuild bot with web SPA:
ansible-playbook playbooks/deploy_tma.yml \
  -e "tma_domain=vpn.example.com tma_certbot_email=admin@example.com"

# Or include in full stack:
ansible-playbook playbooks/stack.yml \
  -e "bot_telegram_token=... bot_admin_id=... tma_domain=vpn.example.com tma_certbot_email=admin@example.com"
```

**Variables** (set in `inventory/group_vars/all.yml` or as `-e` overrides):

| Variable | Default | Description |
|----------|---------|-------------|
| `tma_domain` | `""` | Domain for Web App (required to enable TMA) |
| `tma_https_port` | `8444` | HTTPS port (nginx) вҖ” no conflict with XRay:443 |
| `tma_backend_port` | `3000` | Fastify port (localhost only) |
| `tma_certbot_email` | `""` | Let's Encrypt email (required) |

**After deploy:** The bot auto-registers a Menu Button (вҳ°) in Telegram pointing to `https://<tma_domain>:<tma_https_port>`.

**TMA features:**
- Client list with instant search + filters (Active/Suspended, WG/XRay/Both)
- Create client via form вҖ” closes Web App вҶ’ config `.conf` + QR codes arrive in chat
- Suspend / Resume / Delete clients
- Send config to chat on demand
- Full Telegram theme support (light/dark auto-switch)
- Works on all Telegram clients (iOS, Android, Desktop)

### 10. Backup & Restore (recommended)

Back up critical server state (keys, configs, bot SQLite DB) to the local controller:

```bash
# Create a timestamped backup:
ansible-playbook playbooks/backup.yml
# вҶ’ artifacts/backup/2026-03-01T12-00-00/
# вҶ’ artifacts/backup/latest (symlink)
```

Restore from backup (e.g., after server reprovisioning):

```bash
# Restore from latest backup:
ansible-playbook playbooks/restore.yml

# Restore from a specific snapshot:
ansible-playbook playbooks/restore.yml -e "backup_name=2026-03-01T12-00-00"

# Then re-template all derived configs:
ansible-playbook playbooks/stack.yml
```

**What gets backed up:**

| Server | Files |
|--------|-------|
| A | WG private/public keys, `wg-clients.conf` |
| B | Reality private/public keys, shortId, bot SQLite DB (`data.db`) |

All existing client configs continue working after restore (same keys, same UUIDs).

## Firewall Mode (`manage_ufw`)

| Aspect | `manage_ufw: "keep"` | `manage_ufw: "disable"` |
|--------|---------------------|------------------------|
| Firewall | UFW enabled | UFW disabled |
| Rule storage | `/etc/ufw/before.rules` | `iptables-persistent` |
| Rule management | blockinfile + ufw module | ansible.builtin.iptables |
| Persistence | Automatic via UFW | netfilter-persistent save |
| Best for | Existing UFW setups | Clean iptables setups |

**Important:** Never mix both mechanisms. Choose one mode and stick with it.

## Verify Installation

### Cascade checks

```bash
ansible-playbook playbooks/verify_wg_cascade.yml
```

Expected: wg-clients active on Server A, TPROXY fwmark rule present (`ip rule show | grep fwmark`), local route in table 100 (`ip route show table 100`), XRAY_WG_TPROXY mangle chain populated.

### Relay checks

```bash
ansible-playbook playbooks/relay.yml --tags verify
```

Expected: DNAT rules in PREROUTING, FORWARD ACCEPT rules, UFW status active (keep mode).

### XRay server checks

```bash
ansible-playbook playbooks/verify_xray.yml
```

Expected: xray service active, port 8443 listening, client count displayed.

### Memory/swap checks

The verify playbooks emit warnings when:
- `MemAvailable < 128 MB` вҖ” risk of OOM
- `SwapTotal == 0` вҖ” no swap configured

Manual check:
```bash
free -h
grep -E 'MemAvailable|SwapTotal|SwapFree' /proc/meminfo
```

### All-in-one verify

```bash
ansible-playbook playbooks/verify_all.yml
```

All port numbers (`xray_port`, `port_a_tcp`, `port_b_tcp`, `wg_clients_port`, `xray_tproxy_port`, `xray_tproxy_table`) are defined in `all.yml` as the single source of truth. Per-group files override only non-port settings.

## Rollback

```bash
# Cascade: stop WireGuard, remove config (keeps keys by default)
ansible-playbook playbooks/rollback_wg_cascade.yml

# Cascade: also delete key material
ansible-playbook playbooks/rollback_wg_cascade.yml -e "wg_cascade_remove_keys=true"

# Relay: remove all relay config
ansible-playbook playbooks/rollback.yml -e "manage_ufw=keep"

# Relay (disable mode): specify ip for rule removal
ansible-playbook playbooks/rollback.yml -e "manage_ufw=disable"

# XRay: stop service, remove config (keeps keys by default)
ansible-playbook playbooks/rollback_xray.yml

# XRay: also delete key material
ansible-playbook playbooks/rollback_xray.yml -e "xray_remove_keys=true"
```

## DPI Evasion Notes (Russia / TSPU)

Empirically tested findings вҖ” update this section when behaviour changes.

### SNI is the primary DPI bypass factor

Russian TSPU specifically blocks VLESS+Reality on port 443 when the SNI matches
well-known Reality camouflage domains. `www.microsoft.com` is catalogued and
blocked. `www.googletagmanager.com` is not (as of 2026-02).

| SNI | Port | From Russia | Notes |
|-----|------|-------------|-------|
| `www.microsoft.com` | 443 | вқҢ | DPI drops TLS data after TCP handshake вҖ” "failed to read client hello" |
| `www.googletagmanager.com` | 443 | вң… | Works вҖ” DPI sees legitimate Google traffic |
| `www.googletagmanager.com` | 8443 | вң… | Works вҖ” less DPI scrutiny + good SNI |

**Current defaults use `www.googletagmanager.com:8443`** вҖ” double protection: if
the SNI gets catalogued, port 8443 still receives less scrutiny than 443; if
8443 gets flagged as non-standard HTTPS, the SNI camouflage still holds.

### How the relay bypasses DPI for Russian clients

Direct connection (client in Russia вҶ’ Server B abroad) is filtered on port 443
with common SNIs. The relay path works because:

1. Client вҶ’ Server A `:443` вҖ” looks like normal HTTPS to a domestic IP
2. Server A DNAT forwards raw TCP to Server B `:8443` вҖ” kernel-level, opaque to DPI
3. Server B verifies the Reality handshake; from its view, source is Server A

### XRay TPROXY for WG client traffic (server-side bypass)

Russian ISPs also block all outbound UDP from Russia to foreign servers (TSPU).
This means Server A cannot forward WireGuard traffic to Server B directly.
Instead, **XRay TPROXY** on Server A intercepts all decrypted WG client packets
and forwards them via VLESS+Reality to Server B, preserving the original destination.

Data path:
1. WG client вҶ’ wg-clients :51888/udp (Server A terminates WireGuard here)
2. Decrypted packet (src=10.66.0.x, dst=internet) hits wg-clients interface
3. iptables mangle PREROUTING: `-i wg-clients вҶ’ TPROXY :12345, mark 0x1`
4. ip rule fwmark 0x1 вҶ’ table 100 вҶ’ local delivery to lo
5. XRay TPROXY inbound reads original dst via `followRedirect`
6. XRay VLESS+Reality outbound вҶ’ Server B:8443 (with original dst preserved)
7. XRay freedom on B creates connection to original dst from Server B's IP

**Server B has no WireGuard.** XRay handles application-level NAT.

### VLESS link strategy

Generate two links per client (both in `artifacts/xray/<name>.vless.txt`):

- **`#name`** вҖ” direct to Server B `:8443` вҖ” use from non-Russian locations
- **`#name-via-relay`** вҖ” via Server A `:443` вҖ” use from Russia

### xray version compatibility

Reality protocol handshake is not backward compatible across major versions.
Clients running xray 26.x fail to authenticate against a server running 24.x
("failed to read client hello" / "processed invalid connection").

**Keep server and client xray versions in sync.** Pin `xray_version` in
`roles/xray_server/defaults/main.yml` and update when clients upgrade.

### Diagnosing DPI issues

```bash
# Does the server see the ClientHello at all?
# Enable debug on server-b and watch logs while client connects:
ssh root@<server-b> "journalctl -u xray -f"

# "failed to read client hello" вҶ’ DPI is dropping TLS data before it reaches the server
# "processed invalid connection" вҶ’ connection reaches server, Reality auth fails (wrong keys/SNI)
# "received real certificate" (client-side) вҶ’ server rejected auth, fell back to camouflage site

# Test TCP reachability:
nc -zv <server-b-ip> <port>

# Test Reality fallback (non-VLESS client вҖ” should return camouflage site cert):
openssl s_client -connect <server-b-ip>:<port> -servername <sni>
```
